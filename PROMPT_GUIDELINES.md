# Prompt Guidelines for Grand Prix Pal

Use this page as shared context for AI-assisted edits across the project.

## Purpose
Generate production-ready updates for a Vite + React + TypeScript travel-and-racing content site, with SEO, accessibility, and performance treated as core requirements.

## Project Context
- Frontend: React with TypeScript, Vite, Tailwind utilities, component-based pages.
- Domain: Formula 1 calendar, race pages, planners, experiences, affiliate links, status pages.
- Priority areas: trustworthy content, strong UX on mobile, clean internal linking, and search discoverability.

## Core Output Rules
- Make the smallest safe change that solves the request.
- Preserve existing naming, file structure, and visual language unless asked to redesign.
- Avoid broad refactors when a localized fix is enough.
- Keep code easy to scan: short functions, clear props, minimal nesting.
- Do not invent APIs, data fields, or routes. Reuse existing data modules and helpers.

## Data and Domain Rules
- Race facts, dates, and standings must come from existing project data files or clearly marked placeholders.
- Never present speculative race outcomes as facts.
- Keep country, circuit, and event naming consistent with current site conventions.
- If data is missing, surface a graceful fallback UI instead of failing.

## UI and UX Rules
- Mobile-first layouts by default.
- Maintain good spacing rhythm and visual hierarchy.
- Add loading, empty, and error states for dynamic sections.
- Keep interactions obvious; avoid novelty that harms readability.
- Respect accessibility:
  - Semantic headings in order
  - Keyboard focus visibility
  - Sufficient color contrast
  - Descriptive link and button text

## SEO and Content Rules
- Every page should support unique title, description, and canonical intent.
- Prefer descriptive section headings with clear search intent.
- Keep copy concise, factual, and skimmable.
- Add structured data only when it matches visible page content.
- Avoid duplicate or near-duplicate intro paragraphs across race pages.

## AMP (Accelerated Mobile Pages) Compliance Rules

This project serves static AMP counterparts for all content pages at `/amp/<path>`.
AMP pages are generated by running `node scripts/generate-amp.mjs` (or `npm run amp:generate`).
The `usePageMeta` hook automatically injects `<link rel="amphtml" href="/amp<path>">` at runtime.

### When adding a new page
1. Add a new page-builder function (e.g. `myNewPage()`) in `scripts/generate-amp.mjs` following
   the existing pattern: call `ampPage({ path, title, description, bodyHtml })`.
2. Register it in the `buildPageManifest()` array with the correct `outPath`.
3. Run `npm run amp:generate` to regenerate all AMP files before committing.
4. If the new page should NOT have an AMP counterpart (e.g. redirect-only or auth pages),
   pass `amp={false}` to `usePageMeta`.

### AMP document rules (enforced by Google's validator)
- First child of `<head>` must be `<meta charset="utf-8">`.
- `<html ⚡>` or `<html amp>` is required on the root element.
- Include `<script async src="https://cdn.ampproject.org/v0.js"></script>`.
- Include the exact AMP boilerplate `<style amp-boilerplate>` block (do not modify).
- All CSS lives inside a single `<style amp-custom>` block — no external stylesheets.
- `<style amp-custom>` must not exceed 75 KB.
- No inline `style=""` attributes (use CSS classes inside `amp-custom` instead).
- No custom JavaScript (`<script>` tags other than AMP runtime are forbidden).
- All images must use `<amp-img>` with explicit `width` and `height` attributes.
  Add `<script async custom-element="amp-img" src="...">` if needed.
- Interactive components (carousels, accordions, etc.) must use approved AMP components
  (e.g. `<amp-accordion>`, `<amp-carousel>`) declared with their own `<script>` tags.
- Every AMP page must have `<link rel="canonical" href="https://grandprixpal.com/<path>">` pointing
  to the canonical React page.
- Every canonical page must have `<link rel="amphtml" href="https://grandprixpal.com/amp/<path>">`
  (handled automatically by `usePageMeta` — do not add manually).

### Content rules for AMP pages
- AMP pages are simplified but must faithfully represent the canonical page's core content.
- No placeholder text — all content must match what is visible on the canonical page.
- Keep the page title and meta description identical to the canonical `usePageMeta` call.
- Do not add affiliate tracking scripts or PostHog to AMP pages — AMP analytics requires
  `<amp-analytics>` with an approved vendor; add only if a validated config exists.

### Validation
- Validate AMP pages using the [AMP Validator](https://validator.ampproject.org/) or the
  browser extension before shipping.
- CI should run `npm run amp:generate` to keep `/public/amp/` in sync with data changes.

## Performance Rules
- Minimize unnecessary re-renders and avoid heavy client-side computation in render paths.
- Lazy-load non-critical UI where appropriate.
- Keep bundle impact in mind when adding dependencies.
- Prefer existing utility functions before creating new abstractions.

## Testing and Validation Rules
- For logic changes, include or update focused tests when test coverage exists nearby.
- Verify TypeScript and lint compatibility.
- Confirm no regressions in:
  - route rendering
  - primary navigation
  - key race/planner pages
  - affiliate/link tracking behavior where touched

## Preferred Response Format from AI
- Start with a short summary of what changed.
- List exact files touched.
- Explain key decisions and tradeoffs briefly.
- Call out anything not validated (for example: tests not run).
- Suggest 1 to 3 practical next steps only when useful.

## Guardrails
- No fabricated metrics, testimonials, or legal claims.
- No placeholder lorem ipsum in final content.
- No destructive edits outside requested scope.
- Flag uncertainty explicitly instead of guessing.

## Definition of Done
A task is done when:
- Requested behavior/content is implemented.
- Existing behavior is preserved outside scope.
- Accessibility and SEO basics remain intact.
- Code is type-safe and consistent with project patterns.
- Any limitations or assumptions are clearly documented.
- **New pages**: an AMP counterpart has been added to `scripts/generate-amp.mjs` and
  `npm run amp:generate` has been re-run (or the page explicitly opts out with `amp={false}`).
- **Data changes**: AMP pages have been regenerated so they stay in sync with canonical content.
